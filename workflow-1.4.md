# 2026 独立开发者 AI 全栈工作流 (v1.4)

## 核心原则

### 1. 批判式需求梳理

别直接写需求，先让 AI 当**“挑刺专家”**。告诉它你要做什么，但命令它先别帮你整理文档，而是先抛出几个最容易翻车的边界场景——比如“AI 接口限流了怎么办”“Markdown 里塞超大图片会不会把数据库撑炸”。把这些坑一个个聊透、确认之后，再让它把最终的 `prd.md` 写出来。这样产出的文档会比顺着你的想法写出来的东西密实得多，后期返工也少。

### 2. 数据驱动技术选型

大模型随口推荐的技术栈，不能直接当真。任何第三方依赖，都必须先让它拉一份**硬数据**出来对比：npm 下载量、包体积、最后更新时间、兼容性、已知安全问题。摆出指标，再由它给出唯一推荐，并把理由写进 `architecture.md`。**直觉靠不住，数字说了算。**

### 3. 分层注入，按需索取

不要把所有规范文件一股脑儿塞进每一次请求里，那样既费 token，又容易让 AI 迷失在细节里。换成三层策略：

* **第一层：极简系统指令**。在 `CLAUDE.md` 里只写下规范文件的路径（`prd.md`, `architecture.md`, `tasks.md`），并加一句强制指令：“**执行任何涉及该模块的任务前，必须先用文件读取工具完整读取最新规范文件，禁止依赖你内部的记忆。**”
* **第二层：动态任务上下文**。开始一个新子任务时，只从规范文件里提取**与当前任务直接相关的章节**，作为当次对话的上下文前缀。精确、轻量，不挤占推理空间。
* **第三层：稳定基础冻结**。那些几乎不会变的东西（全局 TypeScript 版本、命名规范）固定在系统提示里，严格限制长度，项目稳定后尽量不动。

### 4. 契约先行，前后端解耦

参照 OpenSpec 的思路，先让 AI 定好前后端交互的 JSON 契约，然后基于这份契约生成本地 Mock Server。前端对着 Mock 数据独立开工，两边完全并行，谁也不等谁。

### 5. 贯彻TDD

测试**不追求覆盖率数字**，只盯住最容易出问题的核心状态管理和复杂算法。同时用 Husky 的 `pre-commit` 钩子在本机卡住明显有问题的代码，避免白白消耗云端 CI 的算力额度。

### 6. 最小化纠偏，一旦发现即刻干预

当 AI 第一次表现出某种模式性错误（如忘记运行格式化，或跳过类型检查），就立刻往 `CLAUDE.md` 里面追加一条极简规则。**不等它反复犯错才出手——第一次就掐住。**

### 7. 用“困惑”当作架构探针

在 `CLAUDE.md` 里埋一句：“如果你在阅读代码时感到困惑，请将你的困惑点更新到这个文件中。”如果有一天 AI 真的跑来说“这个模块我看不懂”——**不要让它补文档，而是去重构那部分代码**。连 AI 都要绕半天的逻辑，人早晚也会踩进去。

### 8. 战略性卸下思想包袱

遇到 AI 过度谨慎、脑补出“向后兼容”等冗余逻辑时，直接告诉它：“这是一个完全没有真实用户的绿地项目，不用考虑向后兼容，数据结构可以随意调整。”卸掉负担后，它往往能给出最干净直接的实现。

### 9. 跨步推导，强行绕过死锁

当 AI 实现多步骤功能在中间步骤卡死、陷入死循环时，别死磕。**直接命令它去生成最终步骤的输出**。模型的自回归特性会迫使它内部寻找另一条路径闭环逻辑。（注：事后需人工校验一下接口是否对齐）。

## 工具与分工矩阵

| 工具 / 平台 | 核心定位 | 核心职责与应用场景 |
| :--- | :--- | :--- |
| **DeepSeek v4 Pro** | 主力输出 | 负责需求对抗、技术选型评估、任务拆解。携带任务上下文生成核心业务代码和针对性测试。 |
| **DeepSeek Flash** | 轻量助手 | 处理超长报错日志分析、写正则表达式、根据接口契约生成 Mock 数据等高频轻量任务。 |
| **Claude Code CLI** | 迭代引擎 | 拥有本地文件读写权限。负责把生成的代码集成进项目、运行测试、反馈错误、驱动多轮修复循环并清理上下文。 |
| **Gemini Pro** | 视觉到代码 | 输入设计稿截图，输出提取了 Design Tokens、用 Tailwind 排版好的单文件静态 HTML。 |
| **GitHub Actions / Harness** | 线上安全网 | 初期仅做 lint、test、build 和预览部署；后期（有用户量后）引入金丝雀发布与自动回滚。 |

## 实际工作流程

### 第 0 步：需求反堵，先清坑再定型

跟 DeepSeek v4 Pro 发起一次对抗式对话，产出一份经过反复拷打的需求文档，存为 `prd.md`。
> “你现在是一位有 10 年经验的产品专家，专门负责挑毛病。我准备开发一个极客博客。请你先别帮我写文档，先抛出 5 个最可能翻车的业务边界或异常场景（比如 AI 接口限流、Markdown 长图撑爆数据库这类），我们逐个讨论，确认后再帮我输出最终的 `prd.md`。”

### 第 1 步：用数据选型，搭建架构契约

确定技术选型后，把理由写入 `architecture.md`，同时生成一份极简的 `CLAUDE.md`，并让 Claude Code CLI 生成本地 Mock Server。此步结束记得执行 `/clear` 清理冗长对话。
> “关于博客的 Markdown 解析器，不要直接给我推荐。先列出 Node 生态中近一年下载量最高的三个库，对比它们的包体积、最后更新日期、Vue 3 兼容性以及 XSS 防护能力，然后给出唯一推荐。”

### 第 2 步：任务拆分与依赖编排

让 DeepSeek v4 Pro 担当项目经理，产出唯一执行清单 `tasks.md`，防止陷入深挖死循环。
> “你现在是一名资深技术项目经理。请根据 `prd.md` 和 `architecture.md`，将这个项目拆分为多个可独立开发、测试和集成的子任务。要求：
>
> 1. 每个子任务产出可验收的明确结果。
> 2. 标注任务之间的依赖关系和推荐执行顺序。
> 3. 将每个任务与推荐使用的 AI 工具匹配。
> 4. 输出一份 `tasks.md`，格式清晰，便于后续逐步执行。”

### 第 3 步：设计稿转静态原型

把参考截图丢给 Gemini Pro，将返回文件保存为 `prototype.html` 作为视觉参照。
> “提取这张截图的 Design Tokens，用 Tailwind CSS 还原排版，输出一个单文件 index.html。”

### 第 4 步：按任务迭代，生成代码与测试闭环

对着 `tasks.md` 里的子任务逐一推进：

#### 4.1 准备任务上下文

从规范文件中提取与当前子任务**直接相关**的规则与契约。必要时声明“这是无用户的绿地项目，不考虑向后兼容”。

#### 4.2 生成初始实现与测试

让 DeepSeek v4 Pro 基于上下文生成临时文件（如 `temp_feature.*`）：
> “你是资深全栈，请基于当前任务要求，为『这个子模块』编写 5 个 Vitest 测试用例，覆盖极端边界情况；然后给出能使这些测试全部通过的 Vue 3 / TypeScript 业务代码。一次性输出代码与测试。”

#### 4.3 本地集成与首轮验证

让 Claude Code CLI 将临时代码集成到正确位置，依次运行：

```bash
npx eslint --fix   # 静态检查
npx tsc --noEmit   # 类型检查
npm run test       # 测试执行
```

#### 4.4 多轮“报错-修复”迭代

* 如果失败，把完整报错喂给 DeepSeek v4 Pro 修正，再由 Claude Code CLI 覆盖并重试。通常 2~4 轮即可全绿。
* 若某步骤反复卡壳，启用**“跨步推导”**：要求 AI 越过阻塞直接输出最终代码，事后手动校验。
* 若发现 AI 再次犯同类错误，立即将规则追加到 `CLAUDE.md`。
* 全部测试通过后，执行一次 `/clear` 清理对话记忆。

### 第 5 步：本地巩固与提交

交由 Claude Code CLI 负责收尾：

* 跑一次全量单元测试，确认新代码未破坏已有功能。
* 清理临时文件、未使用的导入和调试日志。
* **探针触发应对**：若 AI 主动提议“某块代码让人困惑，建议补充文档”，**转身去重构那部分代码**。
* 通过 `git add` 暂存变更准备提交，此时本地 Husky 钩子会自动执行轻量验证（lint+测试）。

### 第 6 步：云端交付与观察

推送代码后，GitHub Actions 接管：

* **初期模式（无真实用户）**：仅运行验证与构建，部署到预览环境供人工验证。
* **后期模式（有真实用户）**：引入 Harness 等平台，执行安全扫描与生产构建，走金丝雀环境、API 冒烟测试无误后全量发布；异常则秒级回滚。
* **闭环反馈**：CI/CD 中的失败日志自动发送给 DeepSeek Flash，生成根因分析与修复建议，以 Issue/PR 评论自动回传。
