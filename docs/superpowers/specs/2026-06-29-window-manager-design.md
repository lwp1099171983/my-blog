# 阶段 4：窗口管理器设计

日期：2026-06-29

## 目标

补齐博客首页桌面隐喻的窗口交互，让窗口在桌面端可拖拽、可聚焦、可最小化/最大化/关闭，并在移动端保持可用。

阶段 4 采用“完整桌面感”方案：比任务清单多补一点基础体验，但不做位置持久化、双击最大化或复杂窗口状态机。

## 范围

只修改 `public/scripts/window-manager.js`。

保留现有全局函数接口，避免改动 Astro 组件中的 inline handler：

- `focusWindow(id)`
- `minWindow(id)`
- `maxWindow(id)`
- `closeWindow(id)`
- `toggleStartMenu()`

## 窗口行为

- `focusWindow(id)`：显示窗口、设为 `.active`、提升 `z-index`、同步任务栏 active。
- `minWindow(id)`：隐藏窗口，若窗口处于 active 状态则清理任务栏 active。
- `maxWindow(id)`：切换 `.maximized`，并聚焦该窗口。
- `closeWindow(id)`：隐藏窗口、移除 active，若还有可见窗口则激活层级最高的窗口。
- `getNextZIndex()`：读取当前所有窗口的最高层级并加一。
- Escape：关闭当前 `.window.active`。

任务栏保持静态标签，不新增动态标签生成。当前首页默认窗口已由静态标签覆盖，阶段 4 不引入额外状态。

## 拖拽行为

- 只允许从 `.titlebar` 开始拖拽。
- 标题栏按钮不会触发拖拽。
- `.maximized` 窗口不可拖拽。
- `≤768px` 禁用拖拽，使用响应式流式布局。
- 鼠标和触摸共用同一套拖拽状态，减少重复逻辑。
- 拖动时限制窗口不完全跑出视口：至少保留一段标题栏区域，避免窗口丢失。
- 触摸拖拽时只在拖拽过程中阻止默认滚动，避免误伤普通移动端滚动。

## 响应式与动效

- `focusWindow()` 在 `≤768px` 下调用 `scrollIntoView()`。
- 若 `prefers-reduced-motion: reduce`，滚动行为使用 `auto`。
- 否则滚动行为使用 `smooth`。

## 不做

- 不做窗口位置持久化。
- 不做双击标题栏最大化。
- 不做动态任务栏标签创建/销毁。
- 不做多窗口布局管理器。

这些能力只有在后续真的需要更像操作系统时再加。

## 验证

视觉层不写测试。阶段 4 用构建和浏览器手测验证：

```bash
pnpm build
pnpm dev
```

手测清单：

- 标题栏可拖拽窗口。
- 窗口不会被拖到完全找不回。
- 最大化窗口不可拖拽。
- 最小化、关闭、最大化正常。
- 点击窗口、桌面图标、任务栏后 active 状态正确。
- Escape 关闭活跃窗口。
- `≤768px` 下禁用拖拽，点击入口会滚动到窗口。
