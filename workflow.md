# 2026 独立开发者 AI 全栈工作流 (v1.5)

## 目录结构与职责规范

- /docs/prd.md：需求边界、异常场景、产品功能范围
- /docs/architecture.md：技术选型对比、硬指标数据、系统架构设计。
- /docs/tasks.md：当前执行清单，任务拆解与优先级。

## 核心原则

### 1. 批判式需求梳理

别直接写需求，先让 AI 当**“挑刺专家”**。告诉它你要做什么，但命令它先别帮你整理文档，而是先抛出几个最容易翻车的边界场景——比如“AI 接口限流了怎么办”“长文本/超大文件传输会不会把数据库撑炸”。把这些坑一个个聊透、确认之后，再让它把最终的 `prd.md` 写出来。这样产出的文档会比顺着你的想法写出来的东西密实得多，后期返工也少。

### 2. 数据驱动技术选型

大模型随口推荐的技术栈，不能直接当真。任何第三方依赖，都必须先让它拉一份**硬数据**出来对比：npm 下载量、包体积、最后更新时间、兼容性、已知安全问题。摆出指标，再由它给出唯一推荐，并把理由写进 `architecture.md`。**直觉靠不住，数字说了算。**

### 3. 分层注入，按需索取

不要把所有规范文件一股脑儿塞进每一次请求里，那样既费 token，又容易让 AI 迷失在细节里。换成三层策略：

**第一层：极简系统指令**。在 `CLAUDE.md` 里只写下规范文件的路径（`prd.md`, `architecture.md`, `tasks.md`），并加一句强制指令：“**执行任何涉及该模块的任务前，必须先用文件读取工具完整读取最新规范文件，禁止依赖你内部的记忆。**”
**第二层：动态任务上下文**。开始一个新子任务时，只从规范文件里提取**与当前任务直接相关的章节**（例如一个接口字段、一小段业务规则），作为当次对话的上下文前缀。
**第三层：稳定基础冻结**。那些几乎不会变的东西（全局 TypeScript 版本、命名规范）固定在系统提示里，严格限制长度，项目稳定后尽量不动。

### 4. 类型驱动，全栈一体化（替代 Mock 联调）

**放弃前后端解耦与 Mock Server 的大厂思维。** 独立全栈开发不需要自己等自己。遇到需要联调的功能，直接利用 Supabase/Firebase 等 BaaS 平台，或使用 Prisma/Drizzle 等 ORM。先让 AI 生成一份全局共享的 `types.ts` 或数据库 Schema。后续的前后端代码全部基于这份**“强类型规约”**进行一体化开发，直接怼真实接口，免去维护 Mock 数据的二次成本。

### 5. 靶向 TDD，拒绝教条

测试**不追求整体覆盖率数字**，视觉层（如 Tailwind 排版、页面布局样式）绝对不写任何测试，全靠肉眼验收。但对于底层核心逻辑、算法、以及极端边缘情况极多的场景（例如：处理 Vue 3 响应式 Proxy 的深层依赖、内存泄漏排查、或者大模型多轮对话的消息历史数组裁剪与 Token 截断算法），必须严格执行**“甩出契约 -> AI 生成极端 Vitest 用例 -> AI 补全业务代码”**的轻量靶向 TDD 模式。

### 6. 最小化纠偏，一旦发现即刻干预

当 AI 第一次表现出某种模式性错误（如忘记运行格式化、跳过类型检查、或在响应式处理中使用了错误的数据解构导致失去响应性），就立刻往 `CLAUDE.md` 里面追加一条极简规则。**不等它反复犯错才出手——第一次就掐住。**

### 7. 用“困惑”当作架构探针

在 `CLAUDE.md` 里埋一句：“如果你在阅读代码时感到困惑，请将你的困惑点更新到这个文件中。”如果有一天 AI 真的跑来说“这个模块我看不懂”——**不要让它补文档，而是立刻去重构那部分代码**。连 AI 都要绕半天的逻辑，人早晚也会踩进去。

### 8. 战略性卸下思想包袱

遇到 AI 过度谨慎、脑补出“向后兼容”“未来扩展性”等冗余逻辑时，直接告诉它：“这是一个完全没有真实用户的绿地项目，不用考虑向后兼容，数据结构和业务逻辑可以随意调整。”卸掉负担后，它往往能给出最干净直接的实现。

### 9. 左右互搏式自动化 Review

**放弃人工逐行拉分支走 PR 的 Code Review。** 人的大脑在面对自己或看着 AI 刚写完的代码时会有思维定势。利用多模型的差异化优势，让编写核心代码的模型（如 DeepSeek v4 Pro）完成输出后，将代码片段交叉喂给另一个模型（如 Claude），让其专职扮演“挑刺安全专家”，寻找死锁、内存泄漏或逻辑盲区。

## 实际工作流程 (v1.5)

### 第 0 步：需求反堵，先清坑再定型

跟 DeepSeek v4 Pro 发起一次对抗式对话，抛出你要做的事情，命令它先不写代码，先抛出 5 个最容易翻车的业务边界或异常场景（如高并发限流、超大文本边界、状态机死循环）。逐个确认解决方案后，让它输出最终的 `prd.md`。

### 第 1 步：类型驱动，锁定单源真相

确定技术选型后，理由写入 `architecture.md`。紧接着，让 AI 根据 `prd.md` 直接输出全栈共享的 `types.ts`（或数据库 Schema 文件）。后续的所有开发全部以此类型文件为“铁律契约”。此步结束记得在终端执行 `/clear` 清理冗长对话。

### 第 2 步：任务拆分与轻量编排

让 DeepSeek v4 Pro 担当项目经理，结合类型和需求，将项目拆分为可独立验证的子任务，输出 `tasks.md`。每一项任务都明确标注验收标准，防止盲目深挖。

### 第 3 步：设计稿一键转静态原型

引入ui/ux pro max skills
把界面截图丢给 Gemini Pro，指令：“提取这张截图的 Design Tokens，用 Tailwind CSS 还原排版，输出一个单文件 index.html。”保存为 `prototype.html`，人工肉眼看一眼确认视觉，不需要编写任何 UI 测试。

### 第 4 步：按任务差异化迭代（核心防线）

对着 `tasks.md` 里的子任务逐一推进，根据任务类型进行分流：

* **分支 A：普通增删改查 / UI 交互页面**
  直接让 DeepSeek v4 Pro 连通上下文一次性输出全栈实现，Claude Code CLI 本地集成，跑通语法与类型检查即可。
* **分支 B：底层核心逻辑 / 复杂状态管理（启动靶向 TDD）**
  1. **生成测试防线：** 喂给 DeepSeek v4 Pro 上下文：“这是系统的核心裁剪/依赖追踪逻辑，请先编写 5 个包含极端边缘场景、Null 值防范及内存安全边界的 Vitest 测试用例。”
  2. **代码实现闭环：** 检查测试用例符合预期后，指令继续：“现在，给出能让这 5 个用例全部通过的 TypeScript 业务实现。”

### 第 5 步：本地集成与左右互搏 Review

让 Claude Code CLI 将临时代码集成到项目正确位置，进入双验证阶段：

```bash
npx eslint --fix   # 1. 静态与代码格式自动修复
npx tsc --noEmit   # 2. 严格类型检查
npm run test       # 3. 靶向测试用例执行
```

**1. 交叉 Review 拦截（左右互搏）**：
若代码涉及核心高危逻辑（如深层 Proxy 状态流转、敏感数据裁剪），开启新对话将代码喂给另一个模型（如 Claude）进行逻辑自查：“你现在是高级安全与逻辑审查专家。不要看语法规范。请指出这段代码在内存边界处理或极端输入下可能存在的 3 个潜在逻辑漏洞。”

**2. 报错修复循环**： 任何一步失败，把报错或漏洞信息喂给原模型修正，直到全绿。若发现 AI 犯了模式性错误，立刻在 CLAUDE.md 中追加一行铁律。全部通过后 /clear。

### 第 6 步：本地拦截，极速交付

1. 跑一次全量单元测试，确认未破坏老功能。
2. 若 AI 触发探针主动提议“某块代码让人困惑，建议补充文档说明”，绝不补文档，转身命令 AI 共同重构这部分过于晦涩的代码。
3. 通过 git add 暂存变更，本地 Husky 的 pre-commit 钩子自动执行最后一次轻量验证。
4. 直接 git push 推送远端。由 Vercel/Netlify 自动接管构建并全量上线，剔除复杂的灰度发布与线上金丝雀网，用最快的速度让真实用户（或自己）在线上环境进行产品验证。
